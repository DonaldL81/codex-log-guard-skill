# 项目逻辑说明

本文说明 Codex 写盘异常检测工具如何运行，包括 GUI、CLI、Skill、核心流程、数据流和模块职责。

## 1. 总体运行链路

### 1.1 GUI 链路

1. 用户双击 `codex写盘异常检测.vbs`。
2. VBS 隐藏命令行窗口并启动 `CodexLogGuard.ps1`。
3. GUI 初始化路径、状态、窗口控件和事件。
4. 窗口打开后刷新 Codex 日志状态。
5. 用户通过按钮安装/卸载拦截器、开始/停止监测、清理文件、打开目录。
6. GUI 调用 `lib/CodexLogGuardCore.psm1` 中的核心函数。
7. 监测样本写入 `monitor-logs/`，同时展示到界面。

### 1.2 Skill/CLI 链路

1. Codex 读取 `SKILL.md`。
2. 根据用户自然语言意图选择 CLI 命令。
3. `tools/CodexLogGuardCli.ps1` 初始化路径并加载 `lib`。
4. CLI 执行状态检查、固定时长监测、安装/卸载、延迟清理、清空备份或自检。
5. JSON 模式输出给 Codex 总结；非 JSON 模式输出给人类调试。

### 1.3 延迟清理链路

1. Skill 或用户运行 `deferred-clean`。
2. CLI 启动独立 PowerShell 窗口运行 `CodexLogGuardDeferredClean.ps1`。
3. 助手提示用户完全退出 Codex。
4. 检测到 Codex 进程退出后，移动 `logs_2.sqlite*` 到 `logs_backup`。
5. 助手提示用户重新打开 Codex。
6. 新日志库和 `logs` 表生成后自动安装拦截器。

## 2. 启动流程

### 2.1 GUI 启动

1. 设置错误策略和输出编码。
2. 计算项目根目录、`monitor-logs`、用户 `.codex`、`logs_2.sqlite`、`logs_backup`。
3. 加载核心模块并初始化上下文。
4. 初始化单实例互斥锁。
5. 如果传入 `-SelfTest`，执行核心自检后退出。
6. 加载 Windows Forms 和 Drawing 程序集。
7. 创建窗口、分组、标签、按钮和监测表格。
8. 注册按钮事件、窗口关闭事件和定时器。
9. 刷新状态并进入消息循环。

### 2.2 CLI 启动

1. 参数 `Command` 必须属于 `ValidateSet`。
2. 设置 UTF-8 输出编码。
3. 计算项目根目录和 Codex 路径。
4. 加载核心模块并初始化上下文。
5. 根据命令分支执行动作。
6. `-Json` 模式输出结构化 JSON，否则输出简短文本。

## 3. 配置读取逻辑

当前没有独立配置文件。

| 配置来源 | 说明 |
|---|---|
| 脚本默认值 | 默认监测 120 秒、采样间隔 5 秒 |
| 环境变量 | 使用 `$env:USERPROFILE` 定位 `.codex` |
| 项目相对路径 | 使用项目目录下 `monitor-logs` 保存监测记录 |
| SQLite 状态 | 从 `logs_2.sqlite` 判断拦截器、行数、计数和文件大小 |

GUI 的 `guard-settings.json` 属于本地运行状态，已被 `.gitignore` 排除。

## 4. 认证和权限逻辑

不适用。本工具不登录任何服务，不读取 API Key，不保存 Token。它只访问当前用户本机文件和本机进程性能计数器。

## 5. 核心业务流程

### 5.1 状态检查

触发入口：

- GUI 启动或操作完成后刷新。
- CLI `status -Json`。

处理步骤：

1. 检查 `logs_2.sqlite`、WAL、SHM 是否存在。
2. 只读读取 `logs` 表行数。
3. 检查 `block_log_inserts` 触发器。
4. 判断未安装、纯拦截器或计数拦截器。
5. 读取计数表和上轮新增计数。
6. 读取日志文件和备份目录大小。
7. 读取 Codex 进程写盘状态和最近会话元信息。

### 5.2 安装或卸载拦截器

安装时：

1. 连接 `logs_2.sqlite`。
2. 确认 `logs` 表存在。
3. 删除已有 `block_log_inserts`。
4. 创建纯拦截触发器：

```sql
CREATE TRIGGER IF NOT EXISTS block_log_inserts
BEFORE INSERT ON logs
BEGIN
    SELECT RAISE(IGNORE);
END;
```

卸载时删除 `block_log_inserts`。工具不修改用户对话正文或 `logs` 表已有内容。

### 5.3 计数拦截器

计数拦截器只应在监测期间临时启用。

1. 开始监测前读取当前拦截器状态。
2. 如果拦截器未安装，只监测真实写盘，不自动安装。
3. 如果拦截器已安装且计数偏好开启，创建或更新 `codex_log_blocker_counter`。
4. 创建带计数更新的触发器。
5. 监测结束后恢复纯拦截器。

关闭计数偏好会清空历史计数，因此已拦截类结论可能不完整。

### 5.4 固定时长监测

触发入口：

- CLI `monitor -DurationSeconds 120 -Json`。
- GUI 开始监测按钮。

处理步骤：

1. 按采样间隔读取 Codex 进程写入速率。
2. 汇总总写入 MB/s。
3. 记录最高写入进程、PID 和写入速率。
4. 读取最近会话标题和工作目录作为辅助信息。
5. 如果计数器启用，读取本次采样拦截和本轮累计拦截。
6. 生成采样记录并写入 CSV。
7. 根据样本计算评估结论。
8. 停止时恢复纯拦截器。
9. 恢复后再次读取当前状态，并作为 `final_status` 返回。

CLI 监测会阻塞当前命令直到固定时长结束。

### 5.5 写盘评估

当前评估分 GUI 和 CLI 两套实现，核心判断一致：

- 有拦截增长时优先给出已拦截类结论。
- 写盘均值或峰值达到阈值时判定偏高或异常。
- 低均值、低峰值且无明显连续写入时判定正常。
- 样本不足或信息不充分时显示观察中或少量偏高。

后续调整阈值时应优先考虑抽到 `lib` 共用。

### 5.6 清理日志

Skill/CLI 清理统一走 `deferred-clean`。

延迟清理助手流程：

1. 打开独立 PowerShell 窗口。
2. 等待用户完全退出 Codex。
3. 检查进程已退出。
4. 清空 `logs_backup` 上一次备份。
5. 移动当前 `logs_2.sqlite`、`logs_2.sqlite-wal`、`logs_2.sqlite-shm` 到 `logs_backup`。
6. 等待用户重新打开 Codex。
7. 等待新 `logs_2.sqlite` 和 `logs` 表生成。
8. 自动安装纯拦截器。

GUI 中的清理按钮也使用核心清理函数，但要求用户先退出 Codex，不强制结束进程。

### 5.7 清空备份历史

只清空 `C:\Users\<user>\.codex\logs_backup\*`，不处理当前 `logs_2.sqlite*`，不要求 Codex 退出。

## 6. 数据流说明

| 数据 | 来源 | 处理位置 | 保存位置 | 输出 |
|---|---|---|---|---|
| 日志库状态 | `logs_2.sqlite` | `Get-LogGuardStatus` | 不保存 | GUI 状态、CLI status |
| 拦截触发器 | `logs_2.sqlite` | `Invoke-LogBlockerAction` | SQLite trigger | 状态和保护效果 |
| 拦截计数 | `codex_log_blocker_counter` | `Invoke-LogCounterAction` | SQLite 计数表 | GUI/CLI 监测结果 |
| 进程写盘 | Windows 性能计数器 | `Get-CodexProcessWriteStats` | CSV 监测文件 | 写盘评估 |
| 会话元信息 | `state_5.sqlite`、`session_index.jsonl` | `Get-RecentThreadContext` | CSV 监测文件 | 辅助定位最近会话 |
| 监测记录 | 采样对象 | GUI/CLI CSV 写入函数 | `monitor-logs/` | 后续分析 |
| 备份日志 | `logs_2.sqlite*` | `Backup-And-ClearLogs` | `logs_backup/` | 清理回退 |

## 7. 命令概览

| 命令 | 作用 | 典型触发 |
|---|---|---|
| `status -Json` | 查看当前状态 | 快速看状态、看保护状态 |
| `monitor -DurationSeconds 120 -Json` | 固定时长诊断监测 | 检查 Codex 有没有问题 |
| `install` | 安装纯拦截器 | 安装拦截器、开启保护 |
| `uninstall` | 卸载拦截器 | 卸载拦截器、关闭保护 |
| `deferred-clean` | 启动延迟清理助手 | 清理日志、关闭 Codex 后自动清理 |
| `clear-backup` | 清空备份历史 | 清空备份历史 |
| `open-gui` | 打开 GUI | 打开监控面板 |
| `self-test` | 自检 | 检查工具是否正常 |

`clean` 已禁用，不作为 CLI 或 Skill 可调用命令。

`monitor` 返回对象包含监测结论、CSV 路径、采样列表和 `final_status`。`final_status` 是监测结束并恢复拦截器后的当前状态，用于让 Codex 同时展示监测结论和最终保护状态。

| 模块或文件 | 职责 |

| `CodexLogGuard.ps1` | GUI 主程序、交互、显示、监测表格 |

| `README.md` | 用户说明 |

## 9. 日志和历史逻辑

本工具自身监测记录写入项目目录：

```text
.\monitor-logs
```

记录包括时间、写盘速率、进程、PID、会话元信息和拦截计数。记录不包括用户消息正文、Codex 回复正文、`logs` 表原始内容、API Key、Token 或认证信息。

运行产物和历史备份不进入版本控制。

## 10. 错误处理逻辑

1. SQLite 或 Python 错误会返回给 GUI/CLI。
2. Codex 未退出时，清理流程提示用户退出，不强制结束进程。
3. 延迟清理等待退出或等待重建超时后停止，并提示用户重试或打开 GUI。
4. 采样失败不应删除文件或扩大处理范围。
5. 辅助状态失败不应读取或输出敏感正文。

## 11. 交互逻辑

GUI 包含五个主要区域：

1. 写盘评估：显示最终结论和开始/停止监测按钮。
2. 防护状态：显示拦截器状态、计数偏好、累计和上轮新增。
3. 文件管理：显示日志文件、日志行数、备份目录和清理入口。
4. 实时监测：显示总写入、最高进程、当前会话和监测日志路径。
5. 监测明细：按时间倒序显示采样记录。

Skill 交互遵循 `SKILL.md`：

- 诊断前提示会等待约 2 分钟。
- 监测结束后总结核心指标。
- 清理时提示用户按独立窗口操作。
- 不直接执行超出安全边界的删除或修改。
