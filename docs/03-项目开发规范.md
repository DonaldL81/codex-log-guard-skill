# 项目开发规范

本文维护本项目必须遵守的开发、运行、测试、发布、安全和文档同步规则。

## 1. 基本原则

1. 保持工具轻量，不引入 Electron、Tauri 或大型依赖，除非重新评估发布形态。
2. GUI、CLI 和 Skill 共用逻辑优先放入 `lib/CodexLogGuardCore.psm1`。
3. 涉及 SQLite 触发器、计数器、清理和删除边界的改动必须保守验证。
4. README 面向用户，SKILL 面向 Codex 调度，docs 面向开发维护，内容不要互相复制。
5. 用户要求开发时可以修改本地文件；只有用户明确要求提交或推送时才提交 Git/GitHub。

## 2. 运行方式

| 场景 | 命令或入口 | 说明 |
|---|---|---|
| GUI 启动 | `codex写盘异常检测.vbs` | 推荐用户入口 |
| GUI 调试 | `powershell -NoProfile -ExecutionPolicy Bypass -STA -File .\CodexLogGuard.ps1` | 显示 GUI |
| GUI 自检 | `powershell -NoProfile -ExecutionPolicy Bypass -File .\CodexLogGuard.ps1 -SelfTest` | 不打开 GUI |
| CLI 自检 | `powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\CodexLogGuardCli.ps1 self-test` | Skill/CLI 基础验证 |
| 延迟清理助手自检 | `powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\CodexLogGuardDeferredClean.ps1 -SelfTest` | 验证助手可加载核心模块 |
| 查看状态 | `powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\CodexLogGuardCli.ps1 status -Json` | 只读状态 |
| 固定监测 | `powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\CodexLogGuardCli.ps1 monitor -DurationSeconds 120 -Json` | 会等待固定时长 |

## 3. 测试和验证

修改后按影响范围执行验证。

| 修改范围 | 必跑验证 |
|---|---|
| 文档 | 显式 UTF-8 读取确认，无过期命令 |
| SKILL 调度 | 检查 `SKILL.md` 与 CLI `ValidateSet` 一致 |
| CLI 或核心逻辑 | CLI 自检 |
| GUI 或核心逻辑 | GUI 自检 |
| 延迟清理助手 | 助手自检 |
| 清理、删除、归档逻辑 | 检查安全边界和允许路径，必要时人工验证 |
| 发布或忽略规则 | `git status`、`git ls-files --others --exclude-standard` |

推荐基础验证：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\CodexLogGuardCli.ps1 self-test
powershell -NoProfile -ExecutionPolicy Bypass -File .\CodexLogGuard.ps1 -SelfTest
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\CodexLogGuardDeferredClean.ps1 -SelfTest
```

确认 `clean` 禁用：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\CodexLogGuardCli.ps1 clean
```

预期结果是参数校验失败，说明 `clean` 不再是可调用命令。

## 4. 发布规则

发布或推送前确认：

1. README 与当前用户入口和命令一致。
2. SKILL 与当前 CLI 命令、执行要求和安全边界一致。
3. `.gitignore` 未漏掉运行日志、CSV、SQLite、备份目录和 `_archive/`。
4. `git ls-files --others --exclude-standard` 不出现本地素材、备份或运行产物。
5. 自检命令通过。
6. 未经用户明确要求，不自动提交或推送。

## 5. 配置和运行时数据

| 类型 | 默认位置 | 版本控制 | 发布包 | 说明 |
|---|---|---|---|---|
| 源码 | 项目根目录、`lib/`、`tools/` | 是 | 是 | `.ps1`、`.psm1`、`.vbs` |
| 用户文档 | `README.md`、`SKILL.md` | 是 | 是 | 发布必需 |
| 开发文档 | `docs/` | 是 | 可随源码发布 | 面向维护 |
| 截图 | `assets/screenshot.png` | 是 | 是 | README 使用 |
| 监测记录 | `monitor-logs/` | 否 | 否 | 本地运行产物 |
| GUI 设置 | `guard-settings.json` | 否 | 否 | 本地状态 |
| Codex 日志库 | `~\.codex\logs_2.sqlite*` | 否 | 否 | 只允许按安全边界处理 |
| 日志备份 | `~\.codex\logs_backup` | 否 | 否 | 清理暂存 |
| 项目归档 | `_archive/` | 否 | 否 | 本地素材和历史备份 |

## 6. 安全边界

本项目允许处理的文件范围必须与 `SKILL.md` 保持一致。

允许移动、删除或清空：

- `C:\Users\<user>\.codex\logs_2.sqlite`
- `C:\Users\<user>\.codex\logs_2.sqlite-wal`
- `C:\Users\<user>\.codex\logs_2.sqlite-shm`
- `C:\Users\<user>\.codex\logs_backup\*`
- 本工具项目目录下的 `monitor-logs\*`

允许修改：

- `logs_2.sqlite` 内与 `block_log_inserts` 触发器相关的对象。
- 监测计数所需的 `codex_log_blocker_counter` 表。

禁止：

1. 删除整个 `.codex` 目录。
2. 修改 `.codex` 下除上述日志相关文件以外的文件。
3. 删除、移动或修改用户项目文件、源码、配置、密钥、聊天记录或其他非日志文件。
4. 清理失败时扩大删除范围。
5. 未经明确要求强制结束 Codex 进程。
6. 读取或输出用户消息正文、Codex 回复正文、API Key、Token。

## 7. 依赖管理

| 依赖 | 用途 | 说明 |
|---|---|---|
| Windows PowerShell | 运行 GUI、CLI、核心模块 | 目标环境为 Windows |
| .NET Windows Forms | GUI | 系统程序集 |
| Python `sqlite3` | SQLite 操作 | 使用 `python` 或 `py -3` |
| Windows CIM/WMI | 写盘速率采样 | `Win32_PerfFormattedData_PerfProc_Process` |

新增依赖前必须评估体积、安装成本、Skill 调度复杂度和发布影响。

## 8. 编码和文件编辑要求

1. 修改中文文档后用 `Get-Content -Encoding UTF8` 检查。
2. 修改 `.ps1`、`.psm1` 时注意 Windows PowerShell 5.1 编码兼容；包含中文的脚本优先保持 UTF-8 BOM。
3. 不使用 PowerShell 管道整文件回写中文 Markdown 或脚本。
4. 小范围改动优先使用补丁。
5. 修改 CLI 命令时同步 `SKILL.md`、`README.md`、`docs/02-项目逻辑说明.md`。

## 9. 文档同步规则

| 变化 | 同步文档 |
|---|---|
| 用户使用方式变化 | README |
| Codex 调度方式变化 | SKILL |
| CLI 命令变化 | README、SKILL、项目逻辑说明 |
| GUI 文案或交互变化 | README、项目逻辑说明 |
| 核心流程变化 | 项目逻辑说明 |
| 删除、安全、隐私边界变化 | SKILL、项目开发规范 |
| 发布规则或运行产物变化 | 项目开发规范 |
| 项目目标或关键决策变化 | 项目上下文 |
| 新踩坑 | 项目经验积累 |

## 10. 应急处理

1. 如果 GUI 异常，优先通过 CLI 或核心逻辑恢复纯拦截器。
2. 如果清理失败，只提示用户退出 Codex 后重试或打开 GUI，不扩大删除范围。
3. 如果怀疑计数拦截器残留，检查 `block_log_inserts` 触发器 SQL 是否只包含 `SELECT RAISE(IGNORE)`。
4. 如果出现中文乱码，停止继续编辑，先确认是终端显示问题还是文件真实损坏。
5. 如果 Windows 性能计数器卡住，优先建议重启系统或稍后重试，不把问题误判为本工具删除或拦截失败。
