# 项目经验积累

本文记录本项目开发和排查中的项目专属经验。跨项目通用规则应沉淀到通用开发规范，不写在这里。

## 1. 文件大小不是判断写盘的可靠依据

`logs_2.sqlite` 主文件大小可能增长不明显，但 WAL 或进程 I/O 仍在写入。

更可靠的判断：

1. 读取 Codex 进程 `IOWriteBytesPersec`。
2. 观察 `logs_2.sqlite-wal`。
3. 在监测期间统计被拦截插入次数。

因此 Skill 中“检查 Codex 有没有问题”默认跑 2 分钟监测，而不是只看静态状态。

## 2. Codex 对话中的清理不能依赖“Codex 已退出后继续执行”

用户在 Codex 对话中要求清理日志时，Codex 自身仍在运行。若要求用户先退出 Codex，再继续执行当前 Skill，逻辑上不可行。

当前方案：

- Skill 清理统一走 `deferred-clean`。
- 独立 PowerShell 助手等待 Codex 退出。
- 清理后等待用户重新打开 Codex。
- 新日志库生成后自动安装拦截器。

CLI `clean` 已禁用，避免误用。

## 3. 清理日志会移走拦截器

拦截器是安装在当前 `logs_2.sqlite` 数据库里的触发器。清理时移动 `logs_2.sqlite*` 后，旧数据库里的触发器也会一起被移走。

因此清理后必须处理重装问题：

- GUI 保持打开且保护开启时，发现新日志库后自动重装。
- 延迟清理助手会等待新日志库生成并自动重装。
- 不应让用户以为清理后原触发器仍在。

## 4. 计数拦截器只适合监测期间使用

计数拦截器会更新 `codex_log_blocker_counter`，因此每次被拦截时仍会有少量 SQLite 写入。

当前约定：

- 平时安装的是纯拦截器。
- 开始监测时，如果拦截器已安装且计数偏好开启，临时启用计数拦截器。
- 停止监测后恢复纯拦截器。
- 关闭计数偏好会清空历史计数。

## 5. 写盘评估需要滚动窗口

单次写盘峰值容易误报。当前评估思路：

- 样本不足时不急于下结论。
- 拦截增长优先显示已拦截类状态。
- 均值、峰值和连续高写入共同参与判断。
- 无明显写入且无拦截增长才判定正常。

后续如果改阈值，应同步 GUI、CLI 和文档。更好的方向是把评估函数抽到 `lib` 共用。

## 6. Windows 性能计数器可能卡住

写盘速率来自：

```powershell
Get-CimInstance Win32_PerfFormattedData_PerfProc_Process
```

经验：

- 系统状态异常时可能短时间不返回。
- GUI 采样可能看起来像卡住。
- 重启系统后通常恢复。
- 后续稳定性优化优先考虑采样超时或降级。

## 7. 会话标题只是辅助定位

工具从 `state_5.sqlite` 和 `session_index.jsonl` 获取最近会话标题、目录和更新时间。

注意：

- 这只能辅助判断最近哪个会话活跃。
- 不能证明某条写盘一定来自该会话。
- 监测结论应以进程写盘和拦截计数为主。

## 8. Skill 文档要短于 README

README 面向用户，可以解释背景、界面和使用方式。SKILL 面向 Codex 调度，应更像操作协议：

- 入口命令。
- 调度规则。
- 执行要求。
- 安全边界。

不要在 SKILL 中重复 README 的长解释；否则模型读取时更容易抓到说明文字，而不是命令映射。

## 9. 公开 `.gitignore` 不写本机私有目录名

存档素材和旧版备份如果只是本机归档，优先放到 `_archive/` 并统一忽略。

规则：

- 公开 `.gitignore` 写通用目录：`_archive/`。
- 不写个人本地素材名、临时目录名或机器专属路径。
- 确认 `git ls-files --others --exclude-standard` 不出现待发布外文件。

## 10. 编码问题要优先止损

PowerShell 5.1 对包含中文的脚本编码敏感。如果出现乱码或脚本解析异常：

1. 先停止继续编辑。
2. 用显式 UTF-8 读取确认文件内容。
3. 区分终端显示乱码和文件真实损坏。
4. 脚本文件优先保持 Windows PowerShell 兼容编码。
